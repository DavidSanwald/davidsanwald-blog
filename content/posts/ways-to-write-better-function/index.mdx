## Chose function names that describe what the function does

If you extract a function, chose a name that describes what the function does. Do not repeat context from its use in the name.

Example:
Consider jsx returned by a React component:

Avoid naming like this

```jsx
function App() {
  return (
    <div className="App">{shouldRenderCasinoAdd(user) && <CasinoAdd />}</div>
  )
}
```

instead

```jsx
function App() {
  return <div className="App">{isLegalAge(user) && <CasinoAdd />}</div>
}
```

Why?

If you look at the components render method, it's obvious that the function's outcome determines whether the <CasinoAdd/> component is rendered or not.
Therefore the function name does not introduce any additional information.
While it's clear right away that the function determines whether the component renders <CasinoAdd/> or not, you cannot know how the function makes this decision.
Moreover, looking at the extracted function, the name shouldRenderCasinoAdd introduces an artificial context, that prevents reuse of the function.

## Use Object Destructuring for Function Parameters to Reduce Artificial Context

Take the following function

```javascript
function getShortName(employee) {
  const { firstName, lastName } = employee
  return `${firstName[0]}. ${lastName}`
}
const myEmployee = { firstName: "Jane", lastName: "Doe" }

console.log(getShortName(myEmployee)) // J. Doe
```

where the parameter naming introduces context that is not present in the actual function.

If the function is now used to calculate the name for an external applicant the parameter name contradicts the actual function input.
While it's possible to choose more general names like `user` or `person`, this still does not fully solve the problem.

By using object destructuring, we can write the function in a more general way, while being more explicit about the input we expect at the same time:

```javascript
function getShortName({firstName, lastName}){
    ...
}
```
